lhasa

纯 FOSS 的 LHA / LZH / PMarc 解码器，单文件自包含静态二进制发布。

lhasa 内部 vendored 自 fragglet/lhasa ——ISC 协议的库与 CLI，作者 Simon Howard。我们不维护解码器； 我们只把它打包成跨平台零依赖的静态二进制。

macOS / Linux / Windows 安装 看四个常用命令

~200 KB 静态二进制 · 任何 Linux / macOS / Windows 都不需 libc / zlib / libiconv。

最新 release · upstream · 安全审计 · 支持的格式 · 用法 · 安装 · 构建审计

一句话。lhasa 是 lha 命令行工具的纯 FOSS 复刻——读 .lzh（LHA）、.lzs（LArc）、 .pma（PMarc）归档，没有历史上 lha 的非自由 再分发包袱。上游是 fragglet/lhasa （ISC，Simon Howard）；本仓 （ljh-sh/lhasa） 是多架构静态二进制分发。CLI 二进制名字故意是 lha（不是 lhasa），所以老脚本 lha x foo.lzh 直接能跑。 谁该用：在没装 1990 年代归档工具的系统上解 30 年前 .lzh 的人；CI 流水线要一个 200 KB、零外部依赖（libz / libiconv / libcurl 都不要）的二进制；要用 liblhasa 在 C 程序里嵌入解码的人。本站是什么： 安装指南（x eget / apt / brew / 源码编译）、 用法导览（四个常用命令 l / x / t / p）、 格式目录（.lzh / .lzs / .pma 三家族）、源码级安全审计 （关于 vendored 解码器）、构建审计（CI 矩阵、SHA256SUMS、可复现性）。本站不是什么：不是 encoder、不是 lzh 格式扩展、不是从零写的 lhasa-rs。

什么是 lhasa?

Lhasa 是 ISC 协议的库 + CLI，用于读 .lzh（LHA）和 .lzs / .pma 归档。故意只做解码——写 .lzh 仍然是原版 lha 二进制的活（从 jca02266/lha 源码 build）。

lhasa（库，liblhasa）能在 C 程序里嵌入： 文件管理器用它做 LZH 预览；CDN / 归档项目用它按需取 .lzh 内容。lhasa 自带的 CLI 二进制起名 lha——项目故意镜像原 LHa for UNIX 命令，让老工具链 调 lha x archive.lzh 直接透明能跑。

使用方法。四组命令 — 列表、解压、测试、打印：

  lha l archive.lzh             # 列表（ls）
  lha x archive.lzh             # 解压
  lha t archive.lzh             # 测试（CRC 校验）
  lha p archive.lzh file        # 打印到标准输出（cat）

安装：x eget ljh-sh/lhasa · 静态二进制，~200 KB，零依赖。

为什么 lhasa 存在（一段话）

jca02266/lha （LHa for UNIX 1.14i 的维护 fork）在它 man page 里带着一个 作者自定义的分发协议——允许源码再分发（带出处），但 纯二进制再分发须先联系原作者。1990 年代这事不是问题，Linux 发行版直接发二进制。2012 年 Debian 把这个包挪出 non-free 之后，fragglet/lhasa（2011 年 Simon Howard 启动，ISC）成了 标准 FOSS 替代。lhasa 能读原版 lha 写的每种变体（lh0..lh7、lhx、 lzs、pma），但不能写新 .lzh 归档——写还是原版 lha 源码 build 的活。

ljh-sh/lhasa 把 ISC 源码打成单一静态二进制——上游 fragglet/lhasa 只发源码 tar 包 + Windows 预编译包， Linux / macOS 用户想要一行命令直接安装（不用 build）的话， 就用本仓。
