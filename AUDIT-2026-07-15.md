# Source-Level Security Audit — vendored lhasa @ 75ed835

This audit reviews `upstream/lhasa/` in the ljh-sh/lhasa repository as
shipped in [v0.6.0.1](https://github.com/ljh-sh/lhasa/releases/tag/v0.6.0.1).
The audited tree is byte-for-byte identical to upstream commit
`75ed835` of [fragglet/lhasa](https://github.com/fragglet/lhasa/tree/75ed835).
No local patches have been applied; `git diff HEAD~1..HEAD -- upstream/lhasa/`
on the vendoring commit is empty.

The audit was conducted by reading the C source under `lib/` and `src/`.
No fuzzing, no coverage instrumentation, no formal verification.

## Severity scale

| Level | Meaning |
|-------|---------|
| **HIGH** | Reachable by attacker-controlled input, real impact (file overwrite, code execution, etc.) |
| **MEDIUM** | Reachable, but harder to trigger or limited impact (denial of service, partial corruption) |
| **LOW** | Reachable in narrow scenarios, or low impact (decoder corruption, edge case) |
| **INFO** | Not a bug, recorded for posterity |

## Summary

| # | Level | Area | Title |
|---|-------|------|-------|
| 1 | **HIGH** | extract | `..` in `header->filename` is **not** collapsed |
| 2 | **HIGH** | symlink | Same as #1, plus symlink target policy ambiguity |
| 3 | MEDIUM | header | `header->filename` length not validated against `level_3_max_header_len` |
| 4 | MEDIUM | symlink | placeholder-mode file permission is 0600 regardless of umask |
| 5 | MEDIUM | safe.c | output filter only allows 0x20–0x7e; modern terminals interpret sequences within that range |
| 6 | LOW | decode | decoder offset can exceed `RING_BUFFER_SIZE`, producing wrong (not OOB) output |
| 7 | LOW | monitor | `block_size` doubling in `lha_decoder_monitor` overflows unsigned int at theoretical extremes |
| 8 | INFO | upstream | 0.6.0 already fixed `-pm2-` `copy_decode[]` overflow + empty-filename skip |
| 9 | INFO | test | fuzz harness exists upstream but not run in CI; no negative-path tests |

## Detailed findings

### #1 (HIGH) — Path traversal via `header->filename`

**Where**

- `lib/lha_file_header.c:854` `collapse_path()` is called on
  `header->path` only at `lib/lha_file_header.c:1048`, never on
  `header->filename`.
- `src/extract.c:46` `file_full_path()` concatenates
  `extract_path + "/" + header->path + header->filename` with only
  leading-`/` stripping on each component.

**Effect**

A malicious archive can write outside `cwd` by setting the
decompressed filename to a path containing `..` segments. Example
from `/home/user` cwd:

| archive `header->path` | archive `header->filename` | extract writes to |
|------------------------|----------------------------|-------------------|
| `subdir`               | `../../../tmp/x`           | `/tmp/x`          |
| `legit`                | `../../etc/passwd`        | `/etc/passwd`     |

`header->path` is normalised by `collapse_path` (so it never
contains `..`), but `header->filename` is taken as-is in
`file_full_path` after only the leading `/` strip.

**Suggested fix** — add `collapse_path` on the filename in
`lha_file_header_read` after `split_header_filename`:

```c
if (header->filename != NULL) {
    collapse_path(header->filename);
}
```

**Severity rationale.** Reachable by user-controlled input
(untrusted `.lzh` extracted from the network). The threat model
matches analogous CVEs in `unrar` (CVE-2018-1000888 family),
`tar` (CVE-2018-1000888), etc. lhasa's `x install lhasa` enables
exactly this scenario.

### #2 (HIGH) — Symlink filename traversal shares root cause

**Where**

- `lib/lha_reader.c:813` invokes
  `lha_arch_symlink(filename, header->symlink_target)`.
- `is_dangerous_symlink` (`lib/lha_reader.c:7xx`) inspects
  `symlink_target` for absolute paths and `..` segments, but not
  the *link name* (the filename).

**Effect**

A malicious archive with a benign-looking
`symlink_target = "/etc/passwd"` is rewritten to a placeholder
(no actual symlink), so the symmetric target defence works.
However, the same `header->filename = "../../../tmp/innocent"`
uncleared path from #1 routes the placeholder file to outside
`cwd`. Combined with subsequent archive entries, this enables
follow-up file writes anywhere `lha x` can reach.

**Suggested fix** — same as #1, plus defence-in-depth
`realpath`-and-cwd-prefix check before any `lha_arch_*` file
syscall in `lib/lha_arch_unix.c`.

### #3 (MEDIUM) — Extended-header chain cap is per-level

**Where**

- `lib/lha_file_header.c:46` `LEVEL_3_MAX_HEADER_LEN = 1 MiB`
  bounds the level-3 chain.
- Levels 0/1/2 have **no equivalent cap**. A hostile 1 MiB+
  extended-header chain can OOM a 32-bit target.

### #4 (MEDIUM) — Symlink placeholder file mode

**Where**

- `lib/lha_reader.c` `extract_placeholder_symlink` calls
  `lha_arch_fopen(filename, -1, -1, 0600)` with mode 0600.
- Real umask value is ignored (`mode_t` is passed through).

**Effect**

In combination with #2, the placeholder file is owner-readable
regardless of umask. With the default umask 022, a subsequent
file overwrite succeeds, leaving the previous file's mode
unchanged — readable by group/world.

### #5 (MEDIUM) — `safe_output` filter band `[0x20, 0x7e)`

**Where**

- `src/safe.c:55`. Bytes outside `[0x20, 0x7e)` are blanked to `?`.

**Effect**

Defends against 1980s–1990s terminal-emulator control protocols
(e.g., `ESC` sequences). Modern terminal emulators and font /
rendering pipelines have added control sequences within the
0x20–0x7e range (vendor extensions, OSC variants). The TODO on
`safe.c:43` already acknowledges this gap. Practical attack
surface is limited to UI spoofing on a terminal that interprets
the crafted sequence; no code execution path.

### #6 (LOW) — Decoder offset out-of-range

**Where**

- `lib/lh_new_decoder.c:464` (`read_offset_code`).
- Range up to `2^15 - 1 = 32767` for `bits=15`, larger than
  `RING_BUFFER_SIZE = 16384`.

**Effect**

`start` underflows unsigned, but the subsequent `% RING_BUFFER_SIZE`
wraps to a valid index. Output is `ringbuf[wrong_position]`
(corrupted decompression), not OOB memory access. No memory
unsafety.

### #7 (LOW) — `lha_decoder_monitor` block-size doubling

**Where**

- `lib/lha_decoder.c:188`. `block_size` is `unsigned int`,
  doubled in a loop until `stream_length / 131072 <= block_size`.

**Effect**

Theoretical overflow to 0 if `stream_length > 2^47`. Realistic
reach requires attacker-controlled `stream_length`, but the
CLI reads `stream_length` from the archive as `uint32_t`,
capped well below 2^47. Non-issue in current call sites.

### #8 (INFO) — 0.6.0 fixes already vendored

- `-pm2-` `copy_decode[]` read overflow (NEWS.md v0.6.0)
- empty-filename member skipping (NEWS.md v0.6.0)

Both already present in vendored HEAD `75ed835`. No action.

### #9 (INFO) — Test coverage

Tests cover `-lh0/-lh1/-lh4/-lh5/-lh6/-lh7/-lhx/-lzs/-lz5/-pm1/-pm2`
for happy-path round-trips on real upstream archives (3 files
exercised in ljh-sh/lhasa CI smoke). Gaps:

- Fuzz harness (`test/fuzzer.c`) exists but is not run continuously.
- No negative-path test for malformed headers
  (e.g. `header_len = 0xFFFFFFFF`, deeply nested symlink loop).
- Post-CRC-failure buffer pollution is not exercised.

## Trust model

### What we **do** know

| Trust claim | Evidence |
|-------------|----------|
| Vendored tree is byte-identical to upstream HEAD | `git diff HEAD~1..HEAD -- upstream/lhasa/` is empty |
| Upstream author identity is `Simon Howard <fraggle@soulsphere.org>` | upstream GitHub commit metadata |
| No outbound network calls anywhere in the code | `grep -rE 'socket\|connect\(' upstream/lhasa/` returns 0 hits |
| No `system()` / `exec()` / `popen()` outside two test helpers | `test/ghost-tester.c` (test), `gencov` (coverage) |
| No reads of `.ssh`, `.aws`, `/etc/passwd`, etc. | `grep -rE '\.ssh\|\.aws\|\.npmrc\|/etc/(passwd\|shadow)' upstream/lhasa/` returns 0 hits |
| No `__attribute__((constructor))` / `__attribute__((destructor))` | `grep -rE '__attribute__\(\(constructor\|destructor' upstream/lhasa/` returns 0 hits |
| No `getenv()` outside test fixture override | `getenv("TEST_NOW_TIME")` in `src/list.c` |
| lhasa is in Debian `main` (not non-free) | passed Debian license + code review chain |

### What we **do not** know (limiting factors)

| Limitation | Implication |
|------------|-------------|
| Upstream maintainer is **not** GPG-signing releases | A compromised `fragglet/lhasa` repo would not be detected by `git verify-commit` |
| No reproducible build | We cannot independently re-derive the SHA256 of a release from source — we only trust the CI's signed-artifact chain |
| No SAST / fuzz run in CI | Newly introduced regressions would only be caught by human review of lhasa CI outputs |
| Path-traversal findings (#1+#2) are **open** at upstream | `git log fragglet/lhasa master` shows no commits addressing them as of `75ed835` |

## Action plan

1. **File upstream issue for #1** (path traversal in `header->filename`).
   Recommend the `collapse_path(filename)` fix in
   `lha_file_header_read` finaliser.
2. **File upstream issue for #2** (symlink filename traversal).
   Same fix; suggest a `realpath`-prefix check as a second line.
3. **Apply local patch** in `ljh-sh/lhasa/upstream/lhasa/` for #1 + #2
   as an immediate fix, gated via a `git subtree pull --squash`
   rebase when upstream adopts it. Drive the next release as
   `v0.6.0.2`.
4. **Add `LEVEL_1_MAX_HEADER_LEN` / `LEVEL_2_MAX_HEADER_LEN`** (#3)
   in the same patch series.
5. **Defer #5 (safe.c)** until a concrete attack is reported.
